# STARS-Gateway 架构愿景

## 1. 项目背景与动机

传统 Web 应用防火墙 (WAF) 多采用“发现即阻断”的防御模式。然而，面对自动化扫描、慢速 DDoS (Slowloris) 等攻击时，直接阻断不仅会消耗大量资源，还可能为攻击者提供明确的探测反馈。本项目的核心理念是 **“不对称消耗”**：对于可疑流量，系统不返回 403 错误或重置连接，而是接受请求并迫使客户端在无尽等待中耗尽自身资源。

当前 v0.1.0 版本是一个概念验证 (PoC) 原型，实现了单节点、基于 HTTP 协议的慢速陷阱引擎（代号 `STARS-Sink`），验证了“不对称消耗”这一想法的可行性。

本 RFC 旨在描绘项目的中长期技术演进方向。

## 2. 核心设计目标

*   **高性能分流**：将可疑流量的识别与引流下沉至操作系统内核态，实现线速处理。
*   **分布式协同**：构建去中心化的威胁情报共享网络，使单个节点发现的攻击模式能在毫秒级同步至整个集群。
*   **自适应策略**：引入轻量级 AI/ML 机制，根据攻击行为动态调整陷阱策略，实现“道高一尺，魔高一丈”。
*   **最小化依赖与安全隔离**：核心组件应尽量轻量，并通过沙箱等技术确保系统自身安全。

## 3. 架构演进全景图

未来 STARS-Gateway 将演进为一个**分布式、自适应的主动防御网格**。其核心模块将围绕三个层次构建：

# 3.1 高性能数据平面 (STARS-Filter & STARS-Sink)

这是系统的执行层，负责流量的“识别”与“处置”。

*   **STARS-Filter (eBPF/XDP 分类器)**:
*   **功能**：在网卡驱动层（XDP）对入向流量进行初步“评分”。
*   **实现**：利用 eBPF 技术，根据数据包特征（如频率、长度、特定偏移哈希）在内核态完成流量的极速筛选。
*   **动作**：
*   **正常流量**：XDP_PASS，直接上送用户态业务进程，性能损耗极小。
*   **可疑流量**：通过 eBPF Map 重定向至本地陷阱端口（如 TCP/9999），不经过复杂的协议栈处理，实现“零拷贝”引流。
*   **STARS-Sink (用户态陷阱引擎)**:
*   **功能**：负责“处决”STARS-Filter 引流而来的可疑流量。
*   **实现**：基于 Rust + Tokio 构建的异步陷阱引擎。对每个连接实施“引力时间膨胀”——先发送合法的协议响应头，再通过指数退避算法极慢地发送载荷（如每次 1 字节）。
*   **效果**：将攻击方的连接资源长时间挂起，迫使其文件描述符或线程池耗尽，而防御方自身 CPU 占用率极低。

# 3.2 分布式协同层 (STARS-Pulse)

单节点防御能力有限，分布式协同是实现全网免疫的关键。

*   **功能**：去中心化的成员管理与状态同步。
*   **实现**：采用 SWIM (Scalable Weakly-consistent Infection-style Process Group Membership) 协议的变种进行节点发现与故障检测。节点间通过 Gossip（流言）方式传播信息，具有极高的扩展性与容错性。
*   **同步内容**：利用 CRDT (Conflict-Free Replicated Data Types, 无冲突复制数据类型) 数据结构，同步各节点间的“威胁特征哈希”、“IP 信誉评分”等。CRDT 能确保在去中心化环境下，各节点数据最终自动达成一致，无需中央服务器协调。

# 3.3 AI 调度与策略层 (STARS-Pilot)

静态策略无法应对千变万化的攻击，引入 AI 是必然选择。

*   **功能**：安全态势感知、自适应策略生成、行为预测。
*   **实现**：
  . 引入一个轻量级的 AI 模型（如小型 LLM 或强化学习 Agent），负责分析 STARS-Pulse 汇聚的全局威胁态势。
*   **动态策略下发**：AI 调度官根据分析结果，动态调整 STARS-Sink 的陷阱参数（如退避曲线、响应内容），或为 STARS-Filter 生成新的 eBPF 字节码过滤规则。
 *   **安全围栏**：为防止 AI 自身被滥用或出现幻觉，必须设置严格的 Guardrails（护栏机制），确保其生成的策略在沙箱中验证后才可部署。

# 4. 演进路线图

阶段 版本 核心目标 关键交付物
1 v0.2.x 数据平面强化 引入 eBPF/XDP 分类器 (STARS-Filter)，实现内核态透明引流；增强 STARS-Sink，支持多协议陷阱（SSH/MySQL/SMTP）。
2 v0.3.x 分布式协同 实现 STARS-Pulse 模块，基于 SWIM/Gossip 协议完成节点自动发现与 CRDT 状态同步，构建去中心化防御网络。
3 v1.0.0 AI 与自适应 集成 STARS-Pilot 模块，引入轻量级 AI 模型进行全局策略优化与自适应规则生成，形成完整的感知-决策-执行闭环。

# 5. 技术挑战与风险

*   **eBPF 兼容性**：XDP 程序的性能与功能高度依赖网卡驱动和内核版本，需要维护不同内核版本的兼容性。
*   **CRDT 状态同步开销**：在大规模集群中，Gossip 协议和 CRDT 元数据的同步可能带来网络开销，需要精心设计同步策略。
*   **AI 策略的可解释性与安全性**：AI 生成的策略可能难以解释，且存在被对抗样本攻击的风险。必须建立严格的策略审计、沙箱验证和回滚机制。

6. 总结

STARS-Gateway 旨在从单点陷阱工具，演进为具备“感知-决策-执行”能力的自适应、分布式主动防御系统。本 RFC 文档将持续更新，以反映最新的技术探索与社区反馈。
